# 12. Data Management in Distributed Systems

## Managing Data Across Independent Services

---

## 1. The Core Shift

In a monolithic system:

* All modules share a single database
* Data consistency is straightforward
* Transactions are easy to manage

In a distributed system (microservices):

> Each service manages its own data

---

## 2. Database per Service

Each microservice owns its database.

```txt id="db-per-service"
Auth Service → Auth DB  
Order Service → Order DB  
Payment Service → Payment DB  
```

---

### Why this is required

* Services remain independent
* Changes in one service do not impact others
* Enables independent scaling

---

## 3. Problem Introduced: Data Fragmentation

Data is no longer centralized.

Example:

User places an order:

* Order service stores order details
* Payment service stores payment status
* User service stores user information

Data is distributed across services.

---

## 4. Real Scenario: IPL Platform

On JioHotstar:

User activity involves multiple data sources:

* User preferences → User service
* Watch history → Streaming service
* Recommendations → Recommendation service

No single database contains complete information.

---

## 5. Data Duplication

To reduce dependency between services, data is often duplicated.

---

### Example

Order service may store:

* User ID
* Basic user info

Instead of querying user service every time.

---

### Why duplication is used

* Reduces cross-service calls
* Improves performance
* Increases service independence

---

## 6. Trade-Off of Duplication

* Data inconsistency risk
* Updates must be synchronized
* Requires careful design

---

## 7. Eventual Consistency

In distributed systems:

> Data is not updated instantly across all services

---

### Example

User updates profile:

* User service updates immediately
* Other services receive update later

---

### Result

* Temporary inconsistency
* Eventually all services sync

---

## 8. Real Example

On Swiggy:

When an order is placed:

* Order service updates immediately
* Delivery assignment may happen slightly later
* Notifications may be delayed

System remains consistent over time, not instantly.

---

## 9. Transactions in Distributed Systems

In monolith:

* Single transaction ensures consistency

In microservices:

* No global transaction across services

---

### Problem

If one step fails:

* Partial data may exist

---

### Example

Order placed:

* Order saved
* Payment fails

System must handle inconsistency.

---

## 10. Handling Distributed Transactions

Common approach:

* Use event-driven workflows
* Compensating actions

---

### Example

If payment fails:

* Cancel order
* Reverse changes

---

## 11. Key Challenges

### 1. Data Consistency

Maintaining accurate data across services

---

### 2. Synchronization

Keeping services updated

---

### 3. Increased Complexity

More moving parts in system

---

## 12. Key Insight

Distributed data management is not about:

> Keeping everything perfectly consistent

It is about:

> Maintaining acceptable consistency while ensuring scalability

---

## 13. Summary

* Each service owns its data
* Data is distributed across systems
* Duplication is used to reduce dependencies
* Eventual consistency replaces immediate consistency
* Transactions require new approaches
