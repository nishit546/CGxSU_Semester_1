# 06. Service Communication 

## Communication Between Services in Distributed Systems

---

## 1. Distributed System Context

In a microservices architecture, services are:

* Independently deployed
* Running on separate processes or servers
* Communicating over a network

This changes the nature of interaction:

> Communication is no longer in-process function calls, but network-based requests.

---

## 2. REST as the Baseline

Most systems initially use REST over HTTP/1.1 for service communication.

This approach is widely adopted because:

* It is simple and standardized
* It is supported across all platforms
* It is easy to implement and debug

At moderate scale, REST-based communication is sufficient.

---

## 3. Limitations of REST at Scale

As systems grow and the number of inter-service calls increases, REST introduces several limitations.

### 3.1 Latency Accumulation

Each service call introduces network latency.

In a chain of services:

```txt id="latency-chain"
Service A → Service B → Service C
```

The total response time increases with each additional hop.

---

### 3.2 Serialization Overhead

* REST typically uses JSON (text-based format)
* Larger payload sizes
* Increased parsing time

This impacts performance under high load.

---

### 3.3 Connection Inefficiency

HTTP/1.1 handles requests sequentially per connection.

This leads to:

* Increased connection overhead
* Inefficient utilization of network resources

---

## 4. Transport-Level Improvement: HTTP/2

HTTP/2 improves communication efficiency while retaining the HTTP model.

Key features:

* Multiplexing: multiple requests over a single connection
* Header compression
* Reduced latency

This improves performance without requiring major changes to API structure.

---

## 5. High-Performance Communication: gRPC

gRPC is designed for efficient service-to-service communication.

Characteristics:

* Built on HTTP/2
* Uses Protocol Buffers (binary format)
* Supports strongly typed contracts

---

### Advantages

* Lower latency compared to REST
* Smaller payload size
* Efficient for high-throughput systems

---

### Usage Context

gRPC is typically used for:

* Internal communication between services
* Performance-critical paths
* High-frequency data exchange

---

## 6. Real-Time Communication Requirement

Certain system features require real-time updates.

Examples:

* Live score updates
* Chat systems
* Notifications

These use cases are not well-suited for request-response models.

---

## 7. Polling and Long Polling

### Polling

The client repeatedly sends requests at fixed intervals:

```txt id="polling"
GET /updates (every few seconds)
```

Limitations:

* High number of unnecessary requests
* Increased server load

---

### Long Polling

The client sends a request and waits until the server responds.

This reduces unnecessary requests but still maintains a request-response pattern.

---

## 8. Persistent Communication: WebSockets

WebSockets establish a persistent connection between client and server.

Characteristics:

* Bi-directional communication
* Server can push updates without client requests
* Reduced latency for real-time data

---

### Usage Context

WebSockets are suitable for:

* Live updates
* Real-time messaging
* Event-driven user interfaces

---

## 9. Communication Strategy

Different communication methods are used based on system requirements:

* REST → external APIs and general-purpose communication
* gRPC → internal service communication requiring high performance
* WebSockets → real-time updates
* Polling / Long Polling → fallback mechanisms when persistent connections are not feasible

---

## 10. Summary

* Microservices communicate over networks rather than in-process calls
* REST is a common starting point but has limitations at scale
* HTTP/2 improves transport efficiency
* gRPC provides high-performance internal communication
* WebSockets enable real-time interactions
* Communication choice depends on system requirements
