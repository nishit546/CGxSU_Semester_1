## 15.1 Case Study: How Platforms Like JioHotstar Use Docker and Kubernetes

### System Context

During IPL matches (7 PM – 11 PM):

* Millions of users join within minutes
* Traffic spikes are unpredictable
* System must scale instantly and reliably

---

## Step 1: Packaging Services with Docker

Each backend component is packaged as a container using Docker.

Example services:

```txt id="docker-services"
Auth Service → Docker Container  
Streaming Service → Docker Container  
Recommendation Service → Docker Container  
Chat Service → Docker Container  
```

Each container includes:

* Application code
* Runtime (Node.js, Go, etc.)
* Dependencies

---

### Why Docker is Critical

* Same container runs in development and production
* No dependency mismatch
* Fast startup time

---

## Step 2: Deploying Containers on Kubernetes

These containers are deployed on a Kubernetes cluster.

Instead of running manually:

```txt id="manual"
Run 100 containers manually
```

Kubernetes manages:

```txt id="k8s-managed"
Deployments → Pods → Containers
```

---

## Step 3: Handling IPL Traffic Spike

### Before Match (6:50 PM)

* System runs baseline pods:

```txt id="baseline"
Streaming Service → 20 pods  
Auth Service → 10 pods  
```

---

### During Match Start (7:00 PM)

Traffic spikes suddenly.

Kubernetes automatically scales:

```txt id="scale-up"
Streaming Service → 200+ pods  
Auth Service → 50 pods  
```

---

### After Peak (Post Match)

Traffic drops.

Kubernetes scales down:

```txt id="scale-down"
Streaming Service → 30 pods  
```

---

## Step 4: Load Distribution

Incoming requests:

```txt id="lb-k8s"
Users → Load Balancer → Kubernetes Service → Pods
```

Kubernetes ensures:

* Requests are evenly distributed
* No single pod is overloaded

---

## Step 5: Failure Handling (Self-Healing)

If a container crashes:

```txt id="failure"
Pod crashes → Kubernetes detects → New pod created automatically
```

If a machine fails:

* Pods are shifted to another machine

Users experience:

* No visible downtime

---

## Step 6: Rolling Updates During Live Traffic

Example:

* New feature release during IPL

Instead of stopping system:

```txt id="rolling"
Old pods → gradually replaced → new pods
```

This ensures:

* Zero downtime
* Continuous streaming

---

## Step 7: Resource Optimization

Different services scale differently:

* Streaming → heavy scaling
* Recommendations → moderate
* Chat → burst-based scaling

Kubernetes allocates:

* CPU
* Memory

based on demand.

---

## Key Practical Insight

Without Kubernetes:

* Manual scaling → impossible
* Failures → downtime
* Traffic spikes → system crash

With Kubernetes:

> System automatically adapts to real-world load conditions

---

## 14. Summary 

* Orchestration manages containers at scale
* Kubernetes automates deployment, scaling, and recovery
* Pods, deployments, and services form the core structure
* Systems become self-healing and highly available
* Real-world systems dynamically scale based on traffic patterns (e.g., IPL spikes)
