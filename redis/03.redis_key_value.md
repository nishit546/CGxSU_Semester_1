# Working with Key-Value Pairs, Expiration and Persistence in Redis

## Introduction

At its core, Redis is a **key-value store**.

Regardless of whether you work with Strings, Lists, Sets, Hashes, or Sorted Sets, everything in Redis is ultimately stored using a key and a value.

Understanding how keys work is important because every Redis operation starts by identifying a key.

Once you understand key-value storage, you can begin exploring two powerful Redis features:

* Expiration
* Persistence

These features allow Redis to manage temporary data efficiently while still providing options for long-term storage.

---

# Understanding Key-Value Storage

Redis organizes data using a simple concept:

```text
key → value
```

Every piece of information stored in Redis is associated with a unique key.

For example:

```text
student:101 → Arjun
```

Here:

* `student:101` is the key
* `Arjun` is the value

Similarly:

```text
otp:123456 → 987654
```

* `otp:123456` is the key
* `987654` is the value

Redis retrieves data using keys, making data access extremely fast.

---

# Why Are Keys Important?

Keys act as unique identifiers for stored data.

Without keys, Redis would have no way to locate information.

Good key naming also improves readability and maintainability.

Consider the following keys:

```text
user:1001
user:1002
user:1003
```

These keys clearly indicate that the data belongs to users.

Similarly:

```text
student:101
student:102
student:103
```

makes the data easier to understand and organize.

---

# Key Naming Conventions

In professional applications, developers often use prefixes to group related data.

Examples:

```text
student:101
student:102

course:dbms
course:mongodb

product:501
product:502
```

Benefits:

* Better organization
* Easier debugging
* Improved readability

A consistent naming convention becomes increasingly important as applications grow.

# Basic Operations on Key-Value Pairs

Redis provides several commands for working with keys and values.

---

## Storing a Value

Use the `SET` command.

```bash
SET student:101 "Arjun"
```

Output:

```bash
OK
```

Redis stores the value against the specified key.

---

## Retrieving a Value

Use the `GET` command.

```bash
GET student:101
```

Output:

```bash
"Arjun"
```

---

## Updating a Value

Running `SET` again updates the existing value.

```bash
SET student:101 "Krishna"
```

Now:

```bash
GET student:101
```

returns:

```bash
"Krishna"
```

---

## Checking Whether a Key Exists

Use:

```bash
EXISTS student:101
```

Output:

```bash
(integer) 1
```

Meaning:

* 1 → key exists
* 0 → key does not exist

---

## Deleting a Key

Use:

```bash
DEL student:101
```

The key and its value are immediately removed.

---

## Viewing Available Keys

Use:

```bash
KEYS *
```

Example output:

```bash
1) "student:101"
2) "student:102"
3) "course:dbms"
```

This command is useful for learning and debugging.

# What Is Expiration in Redis?

Many types of data are temporary.

Examples include:

* OTPs
* Login sessions
* Password reset tokens
* Cached API responses

Such data should automatically disappear after a certain period.

Redis provides expiration functionality to solve this problem.

Expiration allows Redis to automatically delete data after a specified amount of time.

---

# Why Is Expiration Important?

Consider an OTP verification system.

Suppose an application generates:

```text
OTP: 582941
```

The OTP should only remain valid for 60 seconds.

Without expiration:

* Developers must manually delete old OTPs
* Storage can become cluttered
* Security risks increase

Expiration allows Redis to remove outdated information automatically.

---

# Setting Expiration Using EXPIRE

Suppose we store an OTP:

```bash
SET otp:1001 582941
```

We can make it expire after 60 seconds:

```bash
EXPIRE otp:1001 60
```

Meaning:

* The key exists for 60 seconds
* Redis automatically removes it afterward

---

# Checking Remaining Time with TTL

Redis provides the `TTL` command.

```bash
TTL otp:1001
```

Possible output:

```bash
(integer) 45
```

This means:

* 45 seconds remain before expiration

---

# Creating a Key with Expiration Immediately

Redis provides a shortcut command:

```bash
SETEX otp:1001 60 582941
```

This command:

1. Creates the key
2. Stores the value
3. Sets the expiration

all at once.

This is commonly used in OTP systems.

---

# Practical Examples of Expiration

## OTP Verification

```bash
SETEX otp:101 60 987654
```

OTP expires after one minute.

---

## Login Session

```bash
SETEX session:user101 1800 "logged_in"
```

Session expires after 30 minutes.

---

## Cached API Response

```bash
SETEX weather:ahmedabad 3600 "{...}"
```

Cache expires after one hour.

# What Is Persistence?

Redis is primarily an in-memory database.

This means data is stored in RAM.

A natural question arises:

> What happens if the Redis server restarts?

Without persistence:

* All data stored in memory would be lost.

To address this issue, Redis provides persistence mechanisms.

Persistence allows Redis to save data to disk so that it can be recovered later.

---

# Why Does Redis Need Persistence?

Consider a leaderboard application.

Suppose Redis stores:

```text
Arjun → 950
Mahir → 890
Yashvi → 820
```

If the server suddenly crashes and persistence is disabled:

* All scores disappear.

Persistence ensures important information can be restored after a restart.

---

# Redis Persistence Mechanisms

Redis primarily supports two persistence approaches.

## RDB (Redis Database Snapshot)

RDB periodically creates snapshots of the dataset and saves them to disk.

Think of it as:

> Taking a photograph of the database at regular intervals.

Benefits:

* Fast recovery
* Compact storage
* Suitable for backups

---

## AOF (Append Only File)

AOF records every write operation performed on Redis.

Think of it as:

> Maintaining a log of everything that changes.

Benefits:

* Better durability
* More accurate recovery

---

# Do Developers Always Enable Persistence?

Not necessarily.

The answer depends on the use case.

### Persistence Usually Not Required

* OTPs
* Temporary cache
* Session data

Because losing this information is acceptable.

---

### Persistence Usually Required

* Rankings
* Analytics
* Application data
* Financial information

Because losing this information would be problematic.

---

# Key Takeaways

* Redis stores data using key-value pairs.
* Keys uniquely identify stored information.
* Commands such as `SET`, `GET`, `DEL`, and `EXISTS` are used to manage keys.
* Expiration allows Redis to automatically remove temporary data.
* `EXPIRE`, `TTL`, and `SETEX` are commonly used expiration commands.
* Persistence allows Redis to save data to disk.
* Redis supports both RDB and AOF persistence mechanisms.
