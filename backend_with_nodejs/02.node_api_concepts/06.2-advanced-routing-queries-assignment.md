## Section 1 — Route Parameters

> **What is a route parameter?**
> A route parameter is a dynamic segment inside the URL path itself.
> Format: `/path/:paramName` → read via `req.params.paramName`
>
> Use route params when you are identifying **what** resource to fetch — a category, an ID, a status.

---

### 9. GET `/api/notes/category/:category` — Get by category

Fetch all notes belonging to a specific category. The category name is part of the URL path.

**Example requests:**
```
GET /api/notes/category/work
GET /api/notes/category/personal
GET /api/notes/category/study
```

**Success — `200`:**
```json
{
  "success": true,
  "message": "Notes fetched for category: work",
  "count": 4,
  "data": []
}
```

**Invalid category — `400`:**
```json
{
  "success": false,
  "message": "Invalid category. Allowed: work, personal, study",
  "data": null
}
```

**No notes found — `404`:**
```json
{
  "success": false,
  "message": "No notes found for category: work",
  "data": null
}
```

---

### 10. GET `/api/notes/status/:isPinned` — Get by pinned status

Fetch all notes filtered by pin status. The value in the URL will be the string `"true"` or `"false"`.

**Example requests:**

```
GET /api/notes/status/false
```

**Success — `200`:**

## Section 2 — Query Parameters

> **What is a query parameter?**
> Query parameters come after the `?` in the URL. They are optional and used to filter or shape results without changing the route.
> Format: `/path?key=value&key2=value2` → read via `req.query.key`
>
> Use query params when you are **modifying** the result set — filtering, limiting, adjusting output.

---

### 12. GET `/api/notes/filter` — General filter

Accepts optional query params. If none provided, returns all notes.

**Supported params:**

| Param | Example | Description |
|-------|---------|-------------|
| `category` | `?category=work` | Filter by category |
| `isPinned` | `?isPinned=true` | Filter by pinned status |

**Example requests:**
```
GET /api/notes/filter
GET /api/notes/filter?category=work
GET /api/notes/filter?isPinned=true
GET /api/notes/filter?category=study&isPinned=false
```

**Success — `200`:**
```json
{
  "success": true,
  "message": "Notes fetched successfully",
  "count": 5,
  "data": []
}
```

> Build the filter object dynamically — only add a key if the param exists:
```js
const filter = {};
if (req.query.category) filter.category = req.query.category;
if (req.query.isPinned !== undefined) filter.isPinned = req.query.isPinned === "true";
```

## Section 3 — Pagination

> **What is pagination?**
> Instead of returning all results at once, you return a page of results at a time.
> The client controls which page and how many results per page.
>
> Core formula: `skip = (page - 1) * limit`

---

### 16. GET `/api/notes/paginate` — Paginate all notes

**Supported params:**

| Param | Default | Description |
|-------|---------|-------------|
| `page` | `1` | Which page to return |
| `limit` | `10` | How many results per page |

**Example requests:**
```
GET /api/notes/paginate
GET /api/notes/paginate?page=1&limit=5
GET /api/notes/paginate?page=2&limit=5
GET /api/notes/paginate?page=3&limit=3
```

**Success — `200`:**
```json
{
  "success": true,
  "message": "Notes fetched successfully",
  "data": [],
  "pagination": {
    "total": 21,
    "page": 2,
    "limit": 5,
    "totalPages": 5,
    "hasNextPage": true,
    "hasPrevPage": true
  }
}
```

> Always include `hasNextPage` and `hasPrevPage`:
```js
const page       = parseInt(req.query.page)  || 1;
const limit      = parseInt(req.query.limit) || 10;
const skip       = (page - 1) * limit;
const total      = await Note.countDocuments();
const totalPages = Math.ceil(total / limit);
const notes      = await Note.find().skip(skip).limit(limit);
```

---

### 17. GET `/api/notes/paginate/category/:category` — Paginate by category

Paginate notes within a specific category. Combines a route param with pagination query params.

**Example requests:**
```
GET /api/notes/paginate/category/work?page=1&limit=3
GET /api/notes/paginate/category/study?page=2&limit=5
```

**Success — `200`:**
```json
{
  "success": true,
  "message": "Notes fetched for category: work",
  "data": [],
  "pagination": {
    "total": 8,
    "page": 1,
    "limit": 3,
    "totalPages": 3,
    "hasNextPage": true,
    "hasPrevPage": false
  }
}
```

> `countDocuments` must use the same filter as `find`:
```js
const filter = { category: req.params.category };
const total  = await Note.countDocuments(filter);
const notes  = await Note.find(filter).skip(skip).limit(limit);
```

## Section 4 — Sorting

> **What is sorting?**
> Sorting controls the order results come back in. The client picks which field to sort by and which direction — ascending or descending.

---

### 18. GET `/api/notes/sort` — Sort all notes

**Supported params:**

| Param | Default | Options | Description |
|-------|---------|---------|-------------|
| `sortBy` | `createdAt` | `title`, `createdAt`, `updatedAt`, `category` | Field to sort by |
| `order` | `desc` | `asc`, `desc` | Direction |

**Example requests:**
```
GET /api/notes/sort
GET /api/notes/sort?sortBy=title&order=asc
GET /api/notes/sort?sortBy=createdAt&order=desc
GET /api/notes/sort?sortBy=category&order=asc
GET /api/notes/sort?sortBy=updatedAt&order=desc
```

**Success — `200`:**
```json
{
  "success": true,
  "message": "Notes sorted by title in ascending order",
  "count": 10,
  "data": []
}
```

**Invalid `sortBy` value — `400`:**
```json
{
  "success": false,
  "message": "Invalid sortBy. Allowed: title, createdAt, updatedAt, category",
  "data": null
}
```

> Validate before using — never pass user input directly to `.sort()`:
```js
const allowed = ["title", "createdAt", "updatedAt", "category"];
const sortBy  = req.query.sortBy || "createdAt";
const order   = req.query.order === "asc" ? 1 : -1;

if (!allowed.includes(sortBy)) {
  return res.status(400).json({ ... });
}
const notes = await Note.find().sort({ [sortBy]: order });
```


**Success — `200`:**
```json

  "success": true,
```

## Route Order in note.routes.js

Routes are matched **top to bottom**. Specific routes must come before parameterized ones — otherwise `"bulk"`, `"filter"`, `"sort"` etc. get treated as `:id` values.

```js
// ✅ CORRECT ORDER — specific paths before :id

// CRUD bulk routes first
router.post(  "/bulk", createBulkNotes);
router.delete("/bulk", deleteBulkNotes);

// Route param sections
router.get("/category/:category",          getNotesByCategory);
router.get("/status/:isPinned",            getNotesByStatus);

// Query param sections
router.get("/filter",                      filterNotes);
router.get("/filter/pinned",               getPinnedNotes);
router.get("/filter/category",             filterByCategory);
router.get("/filter/date-range",           filterByDateRange);

// Pagination
router.get("/paginate",                    paginateNotes);
router.get("/paginate/category/:category", paginateByCategory);

// Sorting
router.get("/sort",                        sortNotes);
router.get("/sort/pinned",                 sortPinnedNotes);

// CRUD single-item routes LAST — /:id must always be last
router.post(  "/",    createNote);
router.get(   "/",    getAllNotes);
router.get(   "/:id/summary", getNoteSummary);
router.get(   "/:id", getNoteById);
router.put(   "/:id", replaceNote);
router.patch( "/:id", updateNote);
router.delete("/:id", deleteNote);
```
