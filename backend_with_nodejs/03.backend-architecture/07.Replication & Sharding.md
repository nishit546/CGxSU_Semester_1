# 06. Database Scaling (Replication & Sharding)

## 1. Handling Data at Massive Scale 

Current architecture looks like:

```txt id="current-arch"
Users
  ↓
Load Balancer
  ↓
Multiple Servers
  ↓
Cache
  ↓
Database
```

Everything works… until traffic grows further.

---

## 2. The Real Bottleneck: Database

Even after:

* Horizontal scaling
* Load balancing
* Caching

The database still faces:

* Millions of read requests
* Continuous write operations
* High concurrency

---

## 3. IPL Scenario: Data Pressure

During a live match:

Users are constantly:

* Refreshing scores
* Checking stats
* Placing bets / interactions (if applicable)
* Logging in

This creates:

* Massive reads
* Continuous writes

---

## 4. Why a Single Database Fails

A single database has limits:

* CPU constraints
* Disk I/O limits
* Connection limits

Eventually:

* Queries slow down
* Requests queue up
* System becomes unresponsive

---

# PART 1: REPLICATION (READ SCALING)

---

## 5. What is Replication?

Replication means:

> “Create multiple copies of the same database”

---

## 6. How Replication Works

```txt id="replication-flow"
          → Replica DB 1 (reads)
Primary DB
          → Replica DB 2 (reads)
          → Replica DB 3 (reads)
```

---

### Flow

* Writes → Primary DB
* Reads → Replica DBs

---

## 7. IPL Example

During a match:

* Score updates → written to primary DB
* Millions of users reading score → served by replicas

---

## 8. Benefits of Replication

* Handles massive read traffic
* Reduces load on primary DB
* Improves response time

---

## 9. Hidden Challenge: Replication Lag

Replicas are not always perfectly in sync.

Example:

* Score updated → primary DB
* Replica updates after few milliseconds

User might see:

* Slightly outdated data

---

## 10. When Replication Works Best

* Read-heavy systems
* Content platforms
* Streaming apps

---

# PART 2: SHARDING (WRITE SCALING)

## 11. Why Replication is Not Enough

Replication solves:

* Read problems

But not:

* Write overload
* Massive data size

---

## 12. What is Sharding?

Sharding means:

> “Split data across multiple databases”

---

## 13. How Sharding Works

Instead of one database:

```txt id="sharding-flow"
Users A–M → DB1  
Users N–Z → DB2  
```

Each database stores **only a part of data**

---

## 14. IPL Example

Imagine:

* Millions of users
* Each user generating activity

Instead of:

* One DB handling all writes

We split:

* Region-wise
* User ID range
* Geography

---

## 15. Benefits of Sharding

* Distributes write load
* Handles large datasets
* Improves performance

---

## 16. Hard Part: Choosing Shard Key

Shard key decides:

> “Where does data go?”

Bad choice leads to:

* Uneven load
* Hotspots

Example:

If:

* All active users fall into one shard
  → That shard crashes

---

## 17. Replication vs Sharding (Clear Comparison)

| Aspect     | Replication | Sharding       |
| ---------- | ----------- | -------------- |
| Purpose    | Scale reads | Scale writes   |
| Data       | Same copy   | Split data     |
| Complexity | Medium      | High           |
| Risk       | Data lag    | Data imbalance |

---

# PART 3: COMBINING BOTH (REAL SYSTEMS)

## 18. Real Architecture (IPL Scale)

Platforms like JioHotstar do NOT choose one.

They combine:

* Replication → handle reads
* Sharding → handle writes
* Caching → reduce load
* Load balancing → distribute traffic

---

## 19. Combined Flow

```txt id="final-db-arch"
Users
  ↓
Load Balancer
  ↓
Servers
  ↓
Cache
  ↓
Shard 1 → Replicas
Shard 2 → Replicas
Shard 3 → Replicas
```

---

## 20. Key Insight (Most Important Line)

Scaling databases is not about:

> “Making one DB stronger”

It is about:

> “Distributing data and load intelligently”

---

## 21. Common Mistakes (Real World)

### 1. Scaling too late

System crashes before scaling is applied

---

### 2. Wrong shard key

Uneven traffic → one DB overloaded

---

### 3. Ignoring replication lag

Leads to inconsistent user experience

---

## 22. Conclusion

* Database is the ultimate bottleneck
* Replication solves read scaling
* Sharding solves write scaling
* Real systems use both together
* Data distribution is the hardest problem in backend systems
