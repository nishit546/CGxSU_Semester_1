# 01. Monolithic Architecture

## 1. What is a Monolithic Backend?

A monolithic backend is a **single application** where all parts of a system are built, managed, and deployed together.

Instead of splitting functionality into multiple services, everything exists within one codebase:

* Authentication
* Business logic
* Database operations
* API handling

All components run as part of one unified system.

---

## 2. Real-World Example

Consider an e-commerce backend system.

It typically includes:

* User authentication
* Product management
* Order processing
* Payment handling

In a monolithic architecture:

* All these features are implemented inside **one backend application**
* The application runs on a **single server (or a single deployed unit)**
* All modules usually share the **same database**

---

## 3. How Request Flow Works

Let’s understand how a typical request is handled.

```txt id="flow-1"
Client → Route → Controller → Service → Database → Response
```

Explanation:

* The client sends a request (for example, creating an order)
* The route maps the request to the correct controller
* The controller processes the request and calls the service layer
* The service layer contains the business logic
* The database is used to store or retrieve data
* A response is returned to the client

All of this happens **within the same application boundary**

---

## 4. Key Characteristics of Monolithic Architecture

### 1. Single Codebase

All features are developed and maintained within one project.

---

### 2. Single Deployment

The entire application is deployed as a single unit.

---

### 3. Shared Database

All modules interact with the same database.

---

### 4. Tight Coupling

Different parts of the application are closely connected.

---

## 5. Typical Project Structure (Node.js Example)

```txt id="structure-1"
project/
│
├── controllers/
├── services/
├── models/
├── routes/
├── middlewares/
└── app.js
```

Explanation:

* controllers → handle incoming requests and responses
* services → contain business logic
* models → define database schemas
* routes → define API endpoints
* middlewares → handle authentication, validation, etc.

This structure helps maintain clarity and organization as the application grows.

---

## 6. Why Monolithic Architecture is Widely Used

Monolithic architecture remains a practical choice, especially in early stages of development.

### 1. Faster Development

All components are in one place, making it easier to build and iterate quickly.

---

### 2. Easier Debugging

Since everything runs within a single application, tracking issues is simpler.

---

### 3. Simple Deployment

Only one application needs to be deployed and managed.

---

### 4. Suitable for Small Teams

Small teams can efficiently build and maintain a monolith without dealing with infrastructure complexity.

---

## 7. When Monolith is a Good Choice

Monolithic architecture is well-suited when:

* The application is in its early stages
* The team size is small
* Features are evolving rapidly
* Traffic is manageable

---

## 8. Summary

* A monolithic architecture is a **single, unified backend system**
* All components exist within one codebase and deployment unit
* It is simple to build, test, and maintain in early stages
* It forms the foundation for understanding more advanced architectures
