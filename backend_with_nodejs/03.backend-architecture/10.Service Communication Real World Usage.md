# 10. Service Communication — Real-World Usage

## Part 1: REST and gRPC in Production Systems

---

## 1. Communication in Real Systems

In production systems, communication patterns are not chosen based on theory alone.
They are selected based on:

* Performance requirements
* Type of interaction (internal vs external)
* Frequency of requests
* Real-time requirements

Different communication methods are used for different parts of the system.

---

## 2. REST — External Communication Layer

REST remains the most widely used communication method for **client-facing APIs**.

It is typically used between:

* Mobile applications ↔ backend
* Web applications ↔ backend

---

### Example 1: Swiggy

Feature:

* Fetch restaurant listings
* Place orders
* Track order details

Usage:

* Mobile app sends REST requests to backend APIs
* Example:

  * Fetch restaurants based on location
  * Submit order details

Reason:

* REST is easy to integrate across platforms
* Stateless nature fits user-driven interactions
* Works reliably over public networks

---

### Example 2: Zomato

Feature:

* Search restaurants
* View menus
* Apply filters

Usage:

* Client applications interact with backend using REST APIs

Reason:

* Standardized API design
* Easy to cache responses
* Suitable for request-response workflows

---

### Example 3: JioHotstar

Feature:

* Fetch match details
* Retrieve video metadata
* Load homepage content

Usage:

* Initial data required to render the application is fetched via REST APIs

Reason:

* REST is suitable for structured data retrieval
* Easy integration with web and mobile clients

---

### Key Observation

REST is primarily used for:

* External communication
* User-initiated requests
* Stateless interactions

---

## 3. gRPC — Internal Service Communication

gRPC is commonly used for communication **between backend services**.

It is rarely exposed to clients directly.

---

### Example 1: Google Maps

Feature:

* Route calculation
* Traffic estimation

Usage:

* Routing engine communicates with:

  * Traffic data services
  * Location services

Reason:

* Requires low latency
* High-frequency internal calls
* Efficient binary communication

---

### Example 2: Uber

Feature:

* Matching riders with drivers

Usage:

* Matching service communicates with:

  * Driver location service
  * Availability service

Reason:

* Real-time decision making
* High throughput requirements
* Low latency critical for user experience

---

### Example 3: Netflix

Feature:

* Recommendation engine
* Content metadata retrieval

Usage:

* Backend services communicate with each other to:

  * Fetch user preferences
  * Generate recommendations

Reason:

* High number of internal service calls
* Need for efficient data transfer

---

### Example 4: Swiggy

Feature:

* Order processing and delivery assignment

Usage:

* Order service communicates with:

  * Delivery partner service
  * Tracking service

Reason:

* Fast internal communication
* Handles high request volume during peak hours

---

### Key Observation

gRPC is primarily used for:

* Internal service-to-service communication
* Performance-critical operations
* High-throughput systems

---

## 4. REST vs gRPC — Practical Distinction

```txt id="rest-grpc-summary"
REST → External communication (client-facing APIs)
gRPC → Internal communication (service-to-service)
```

---

### Practical Understanding

* REST prioritizes simplicity and compatibility
* gRPC prioritizes performance and efficiency

Both are used together in modern systems.

## Part 2: WebSockets and Polling in Production Systems

---

## 1. Real-Time Communication Requirements

Not all system interactions follow a request-response pattern.

Certain features require:

* Immediate updates
* Continuous data flow
* Low-latency delivery

Examples include:

* Live sports scores
* Chat systems
* Order tracking
* Financial data updates

In such cases, traditional REST-based communication becomes inefficient.

---

## 2. Limitations of Request-Response for Real-Time Systems

In a request-response model:

* The client must initiate every interaction
* The server cannot send data independently

This creates challenges:

* Delayed updates
* Increased network overhead
* Inefficient resource usage

---

## 3. Polling — Basic Approach to Real-Time

Polling is the simplest method to simulate real-time updates.

### How it works

The client repeatedly sends requests at fixed intervals:

```txt id="polling-flow"
GET /updates (every few seconds)
```

---

### Example 1: Email Systems

Feature:

* Checking for new emails

Usage:

* Client periodically requests inbox updates

Reason:

* Real-time delivery is not critical
* Simplicity is preferred

---

### Example 2: Administrative Dashboards

Feature:

* Monitoring metrics

Usage:

* Dashboard refreshes data at regular intervals

Reason:

* Data does not change every second
* Acceptable delay

---

### Limitations of Polling

* High number of unnecessary requests
* Increased server load
* Inefficient for high-frequency updates

---

## 4. Long Polling — Improved Approach

Long polling reduces unnecessary requests by keeping the connection open.

### How it works

```txt id="long-polling-flow"
Client → Request → Server holds request → Responds when data changes
```

---

### Benefits

* Fewer redundant requests
* More efficient than basic polling

---

### Limitations

* Still based on request-response model
* Connection overhead remains
* Not ideal for very high-frequency updates

---

## 5. WebSockets — Persistent Communication

WebSockets establish a persistent connection between client and server.

### Characteristics

* Bi-directional communication
* Server can push updates without client requests
* Low latency

---

## 6. Real-World Examples of WebSockets

---

### Example 1: WhatsApp

Feature:

* Instant messaging

Usage:

* Messages are delivered in real-time without repeated requests

Reason:

* Requires continuous, low-latency communication
* Immediate delivery is critical

---

### Example 2: Zomato

Feature:

* Live order tracking

Usage:

* Status updates (preparing, out for delivery, delivered) are pushed to users

Reason:

* Users expect real-time visibility
* Frequent updates required

---

### Example 3: TradingView

Feature:

* Live stock price updates

Usage:

* Continuous data stream from server to client

Reason:

* High-frequency updates
* Low latency required

---

### Example 4: JioHotstar

Feature:

* Live score updates
* Real-time engagement (comments, reactions)

Usage:

* Server pushes updates instantly during matches

Reason:

* High concurrency
* Real-time user engagement

---

## 7. WebSockets vs Polling — Practical Comparison

| Aspect        | Polling               | WebSockets            |
| ------------- | --------------------- | --------------------- |
| Communication | Repeated requests     | Persistent connection |
| Efficiency    | Low                   | High                  |
| Latency       | Higher                | Lower                 |
| Server Load   | High                  | Optimized             |
| Use Case      | Low-frequency updates | Real-time systems     |

---

## 8. When to Use Each Approach

### Polling

* Low-frequency updates
* Simpler systems
* Non-critical real-time requirements

---

### Long Polling

* Moderate update frequency
* When WebSockets are not feasible

---

### WebSockets

* Real-time systems
* High-frequency updates
* Interactive applications

---

## 9. Key Observation

Real-time communication requires a shift from:

> Client-driven requests
> to
> Server-driven updates

WebSockets enable this transition effectively.

## Part 3: Communication Strategy and System Design Decisions


## 1. Communication in Real Systems Is Hybrid

Production systems do not rely on a single communication method.

Instead, they combine multiple approaches based on:

* Type of interaction
* Performance requirements
* User experience expectations
* System architecture

Each communication method solves a specific problem.

---

## 2. Mapping Communication Methods to System Layers

A typical large-scale system can be divided into layers:

```txt id="layered-communication"
Client Layer → API Layer → Service Layer → Real-Time Layer
```

Each layer uses different communication strategies.

---

### 2.1 Client Layer (User-Facing)

* Communication: REST
* Purpose: Fetch data, submit requests

Used for:

* Mobile apps
* Web applications

---

### 2.2 API Layer (Gateway / Entry Point)

* Communication: REST
* Purpose: Unified interface for clients

Responsibilities:

* Authentication
* Request routing
* Aggregation of responses

---

### 2.3 Service Layer (Internal Communication)

* Communication: gRPC
* Purpose: Efficient service-to-service interaction

Used for:

* High-frequency internal calls
* Performance-critical workflows

---

### 2.4 Real-Time Layer

* Communication: WebSockets
* Purpose: Continuous updates

Used for:

* Live scores
* Messaging
* Notifications

---

## 3. End-to-End Example: IPL Streaming Platform

On a platform like JioHotstar, a single user action involves multiple communication types.

### Scenario: User Opens a Live Match

---

### Step 1: Initial Data Fetch

* Method: REST
* Purpose: Load match details and metadata

---

### Step 2: Backend Processing

* Method: gRPC
* Purpose: Communication between internal services

Examples:

* Auth service verifies user
* Streaming service prepares content
* Recommendation service fetches suggestions

---

### Step 3: Real-Time Updates

* Method: WebSockets
* Purpose: Push live score updates

---

### Step 4: Background Updates (if needed)

* Method: Polling / Long Polling
* Purpose: Fallback or periodic updates

---

## 4. Decision Factors for Choosing Communication Method

When designing a system, the choice of communication depends on several factors.

---

### 4.1 Latency Requirements

* Low latency required → gRPC or WebSockets
* Moderate latency acceptable → REST

---

### 4.2 Frequency of Updates

* Continuous updates → WebSockets
* Periodic updates → Polling
* On-demand requests → REST

---

### 4.3 System Boundaries

* External communication → REST
* Internal communication → gRPC

---

### 4.4 Complexity vs Simplicity

* Simpler systems → REST
* Complex, high-scale systems → combination of methods

---

## 5. Common Patterns in Modern Architectures

Modern systems follow consistent patterns:

---

### Pattern 1: REST + gRPC Combination

* REST for client communication
* gRPC for internal services

---

### Pattern 2: REST + WebSockets

* REST for initial data
* WebSockets for live updates

---

### Pattern 3: Hybrid Communication

* REST + gRPC + WebSockets used together
* Each solving a different requirement

---

## 6. Common Mistakes in Communication Design

---

### 6.1 Using a Single Method Everywhere

Using only REST for all communication leads to:

* Increased latency
* Inefficient internal communication

---

### 6.2 Overusing Real-Time Systems

Using WebSockets where not required leads to:

* Unnecessary complexity
* Resource overhead

---

### 6.3 Ignoring System Boundaries

Using internal protocols (like gRPC) for external APIs can create compatibility issues.

---

### 6.4 Premature Optimization

Introducing complex communication patterns too early increases system complexity without real benefit.

---

## 7. Key Insight

Effective system design is not about choosing the “best” communication method.

It is about:

> Selecting the appropriate method based on the problem being solved.

---

## 8. Summary

* Modern systems use multiple communication methods together
* REST is used for external APIs
* gRPC is used for internal service communication
* WebSockets enable real-time updates
* Polling is used for simpler or fallback scenarios
* Communication choices depend on latency, frequency, and system boundaries
