# 02. Internal Structure of Monolith

## 1. Why Structure Matters in a Monolith

A monolithic application starts simple.
But as features grow, the codebase can quickly become difficult to manage.

Without proper structure:

* Code becomes hard to read
* Bugs become difficult to trace
* New features take longer to build

A well-structured monolith:

* Keeps logic organized
* Makes development predictable
* Prepares the system for future scaling

---

## 2. Core Idea: Separation of Concerns

A good backend is divided into layers, where each layer has a **clear responsibility**.

Instead of mixing everything together, we separate:

* Request handling
* Business logic
* Data access

This makes the system easier to understand and maintain.

---

## 3. Standard Layered Structure

A common structure in a Node.js monolith looks like:

```txt id="structure-2"
project/
│
├── controllers/
├── services/
├── models/
├── routes/
├── middlewares/
└── app.js
```

This is not random — each folder solves a specific problem.

---

## 4. Understanding Each Layer

### 1. Routes

Routes define the API endpoints.

Example:

```txt id="routes-example"
POST /orders
GET /products
```

They act as the **entry point** of the backend.

---

### 2. Controllers

Controllers handle:

* Incoming request
* Sending response

They should be **thin** and only coordinate actions.

---

### 3. Services

Services contain the **core business logic**.

Example:

* Calculating order total
* Validating payment rules
* Applying discounts

This is where actual application logic lives.

---

### 4. Models

Models define:

* Database structure
* Schema

They are responsible for how data is stored and retrieved.

---

### 5. Middlewares

Middlewares handle:

* Authentication
* Authorization
* Validation
* Logging

They run **before reaching controllers**

---

## 5. Request Flow with Proper Structure

```txt id="flow-2"
Client → Route → Controller → Service → Model → Database → Response
```

This flow ensures:

* Each layer has a clear role
* No mixing of responsibilities

---

## 6. What Happens Without Structure

If you don’t follow separation:

* Business logic inside controllers
* Database queries scattered everywhere
* Duplicate code across files

Result:

* Hard to debug
* Hard to scale
* Hard to maintain

---

## 7. Common Mistakes in Monolith Structure

### 1. Fat Controllers

Controllers doing everything:

* logic
* database
* validation

---

### 2. No Service Layer

Skipping services and directly writing logic in routes/controllers.

---

### 3. Tight Coupling Between Modules

One module directly accessing another module’s internal logic.

---

### 4. No Clear Folder Organization

Everything dumped into a few files.

---

## 8. Better Approach: Feature-Based Structure (Intro)

As the application grows, structure can evolve into:

```txt id="feature-structure"
/modules
   /auth
   /products
   /orders
```

Each module contains:

* its own controller
* its own service
* its own routes

This is called a **modular monolith** (covered in next articles).

---

## 9. Why This Structure is Important

A well-structured monolith:

* Handles growth better
* Reduces code duplication
* Makes team collaboration easier
* Prepares system for future architecture changes

---

## 10. Summary

* Structure is critical for maintaining a monolithic application
* Separation of concerns is the key principle
* Layers like controllers, services, and models keep responsibilities clear
* Poor structure leads to complexity and bugs
