# 03. Problems with Monolith

## 1. When Does a Monolith Start Breaking?

A monolithic application works well in the beginning.

* Few users
* Small team
* Limited features

But as the system grows:

* More users
* More features
* More developers

The same architecture starts showing limitations.

---

## 2. Problem 1: Scaling Becomes Difficult

In a monolith, the entire application runs as a single unit.

If only one part needs scaling (for example, orders):

* You still have to scale the **entire application**

### Example

* Orders API receives high traffic
* Products and Auth are lightly used

Still:

* You must run multiple full instances of the app

This leads to:

* Resource waste
* Inefficient scaling

---

## 3. Problem 2: Tight Coupling Between Modules

In a monolith:

* Modules are closely connected
* One change can affect multiple areas

### Example

* Updating order logic accidentally impacts payment logic
* Small changes require full testing of the system

This increases:

* Risk of bugs
* Development time

---

## 4. Problem 3: Large Codebase

As features grow:

* Codebase becomes large
* Understanding the system becomes difficult

New developers struggle with:

* Finding where logic exists
* Understanding dependencies

This slows down:

* Development
* Onboarding

---

## 5. Problem 4: Slow Deployment

In a monolith:

* You deploy everything together

Even if you change:

* One small feature

You must:

* Build
* Test
* Deploy the entire application

This creates:

* Longer deployment cycles
* Higher risk of failure

---

## 6. Problem 5: Limited Team Scalability

As the team grows:

* Multiple developers work on the same codebase

This leads to:

* Merge conflicts
* Dependency issues
* Coordination overhead

Teams start blocking each other.

---

## 7. Problem 6: Technology Lock-In

In a monolith:

* Entire system uses the same tech stack

You cannot easily:

* Use different technologies for different problems

Example:

* Want Python for AI
* Backend is in Node.js

You are restricted.

---

## 8. Problem 7: Single Point of Failure

If the application crashes:

* Entire system goes down

There is no isolation between components.

---

## 9. Important Insight

Even if you:

* Write clean code
* Follow proper structure

These problems **still appear at scale**

This is not a coding issue —
it is an **architectural limitation**

---

## 10. Summary

* Monolith works well in early stages
* As system grows, limitations appear
* Scaling, deployment, and team coordination become difficult
* These challenges lead to the need for better architectures
