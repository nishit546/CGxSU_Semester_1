# 06. Load Balancing and Caching

## 1. Making Horizontal Scaling Work in Real Systems

* We moved from **one server → multiple servers**
* This solved the **capacity problem**

But it introduced a new challenge:

> “How do incoming requests get distributed across servers?”

---

## 2. Real Scenario: IPL Traffic Spike

During an IPL match on JioHotstar:

* Millions of users click “Watch Live” within seconds
* Each request needs to be handled instantly

Now imagine:

```txt id="problem-flow"
Users → ??? → Server A / Server B / Server C
```

The missing piece is:

> **Who decides which server handles which request?**

---

# PART 1: LOAD BALANCING


## 3. What is Load Balancing?

A **load balancer** sits between users and servers and distributes traffic.

```txt id="lb-flow"
Users → Load Balancer → Server A / Server B / Server C
```

---

## 4. Why Load Balancing is Mandatory

Without load balancing:

```txt id="no-lb"
Users → Server A (overloaded)
       Server B (idle)
       Server C (idle)
```

Result:

* One server crashes
* Others are unused

---

With load balancing:

```txt id="with-lb"
Users → Load Balancer → evenly distributed servers
```

Result:

* All servers are utilized
* No single server overload

---

## 5. How Load Balancer Decides (Simplified)

### 1. Round Robin

* Requests go one by one to each server

### 2. Least Connections

* Sends request to server with least load

### 3. IP Hash (advanced idea)

* Same user → same server

---

## 6. IPL Example (Critical Understanding)

At 7:00 PM:

* 1 million users join

Load balancer ensures:

* ~10,000 users per server (if 100 servers)

Without it:

* First server gets flooded
* System crashes instantly

---

## 7. Hidden Challenge: Stateless Systems

For load balancing to work well:

Servers must be **stateless**

Meaning:

* No user-specific data stored in server memory

Instead:

* Store session/data in shared systems (DB/cache)

---

## 8. New Problem After Load Balancing

Now everything looks stable:

* Servers scaled
* Traffic distributed

But another bottleneck appears:

> “Database is getting overloaded”

---

# PART 2: CACHING

---

## 9. The Real Bottleneck: Database

Even after scaling servers:

Every request:

```txt id="db-hit"
Server → Database → Response
```

During IPL:

* Millions of read requests
* Same data requested repeatedly

Example:

* Match score
* Player stats
* Team info

---

## 10. Why This is a Problem

Database is:

* Slower than memory
* Limited in handling massive concurrent reads

Result:

* Increased latency
* System slowdown

---

## 11. Solution: Caching

Caching means:

> “Store frequently used data in fast memory”

---

## 12. How Caching Works

```txt id="cache-flow"
Request → Cache → (if miss) → Database → Cache → Response
```

---

### Flow Explanation

1. First request:

   * Data not in cache
   * Fetch from database
   * Store in cache

2. Next requests:

   * Directly served from cache

---

## 13. IPL Example

During a live match:

* Millions of users request:

  * Current score
  * Overs
  * Player stats

Without cache:

* DB hit every time → overload

With cache:

* Memory serves most requests

---

## 14. Common Tool Used

* Redis

Why:

* Extremely fast
* Designed for high-read systems

---

## 15. Before vs After Caching

| Scenario    | Without Cache | With Cache |
| ----------- | ------------- | ---------- |
| Speed       | Slow          | Fast       |
| DB Load     | High          | Reduced    |
| Scalability | Limited       | Improved   |

---

## 16. Important Concept: Cache Hit vs Cache Miss

* Cache Hit → Data found in cache
* Cache Miss → Data fetched from DB

Goal:

> Maximize cache hits

---

## 17. Hidden Complexity

Caching is powerful but introduces problems:

* Stale data (outdated info)
* Cache invalidation (when to update?)

Example:

* Score changes → cache must update quickly

---

## 18. Real System Thinking

Platforms like JioHotstar:

* Aggressively use caching
* Optimize read-heavy traffic
* Reduce pressure on databases

---

## 19. Putting It Together

Now your system looks like:

```txt id="final-flow"
Users
  ↓
Load Balancer
  ↓
Multiple Servers
  ↓
Cache (first layer)
  ↓
Database (fallback)
```

---

## 20. Conclusion

Horizontal scaling alone is not enough.

To handle real traffic:

* Load balancing distributes traffic
* Caching reduces repeated work
