# 04. Scaling Basics

## 1. What is Scaling?

Scaling means increasing the system’s ability to handle:

* More users
* More requests
* More data

As an application grows, a single server is often not enough.

Scaling ensures:

* Performance remains stable
* System does not crash under load

---

## 2. Why Scaling is Needed

As traffic increases, systems face problems like:

* Slow response time
* High CPU usage
* Memory exhaustion
* Database overload

Without scaling:

* Users experience delays
* Requests may fail

---

## 3. Vertical Scaling

Vertical scaling means **increasing the power of a single server**.

Example:

* Add more CPU
* Add more RAM
* Upgrade hardware

---

### Characteristics

* Simple to implement
* No architectural changes required

---

### Limitations

* Hardware has limits
* Expensive at scale
* Single machine remains a bottleneck

---

## 4. Horizontal Scaling

Horizontal scaling means **adding more servers instead of upgrading one**.

Example:

* Run multiple instances of the same backend
* Distribute traffic across them

---

### Characteristics

* Better for long-term growth
* Improves system reliability

---

### Challenges

* Requires load balancing
* Requires distributed thinking

---

## 5. Load Balancing

Load balancing distributes incoming requests across multiple servers.

Instead of:

* One server handling everything

We use:

* Multiple servers sharing the load

---

### Basic Flow

```txt id="flow-lb"
Client → Load Balancer → Server 1 / Server 2 / Server 3
```

---

### Benefits

* Prevents server overload
* Improves performance
* Increases availability

---

## 6. Caching (Very Important)

Caching stores frequently accessed data in memory to reduce database load.

Instead of:

* Querying database every time

We:

* Store results in cache

---

### Example

* Product list requested frequently
* Store it in cache
* Serve directly without hitting DB

---

### Benefits

* Faster response time
* Reduced database load

---

### Common Tool

* Redis

---

## 7. Replication

Replication means creating multiple copies of the database.

---

### How it works

* One primary database (writes)
* Multiple replicas (reads)

---

### Benefits

* Improves read performance
* Distributes database load

---

## 8. Sharding

Sharding means splitting data across multiple databases.

---

### Example

Instead of:

* One database with all users

We split:

* Users A–M → DB1
* Users N–Z → DB2

---

### Benefits

* Handles large datasets
* Improves write performance

---

## 9. Key Difference Summary

### Vertical vs Horizontal Scaling

* Vertical → scale **one machine**
* Horizontal → scale **multiple machines**

---

### Replication vs Sharding

* Replication → duplicate data (read scaling)
* Sharding → split data (write scaling)

---

## 10. Important Insight

Scaling is not just about adding servers.

It involves:

* Distributing traffic
* Reducing load
* Managing data efficiently

---

## 11. Summary

* Scaling is required as traffic grows
* Vertical scaling is simple but limited
* Horizontal scaling is powerful but complex
* Load balancing distributes requests
* Caching improves performance
* Replication and sharding handle database scaling

---
