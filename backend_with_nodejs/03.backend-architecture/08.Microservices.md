# 08. Microservices

## Breaking the Monolith for Independent Scaling

---

## 1. The Limitation of a Scaled Monolith

Even after applying:

* Horizontal scaling
* Load balancing
* Caching
* Database scaling

the backend still operates as a **single deployable unit**.

This creates a structural limitation:

> Different parts of the system cannot scale, evolve, or deploy independently.

---

## 2. Real Scenario: IPL Traffic Behavior

During peak hours (7 PM – 11 PM) on platforms like JioHotstar:

Different features experience very different loads:

* Video streaming → extremely high traffic
* Live score updates → high read traffic
* Authentication → moderate
* Recommendations → relatively low
* Chat / engagement → bursty traffic

---

## 3. Problem: Uniform Scaling for Non-Uniform Load

In a monolithic system:

```txt 
Scale Streaming → Entire App Scales
```

This results in:

* Wasted resources
* Higher infrastructure cost
* Inefficient utilization

---

## 4. Problem: Deployment Coupling

A small change in one feature (for example, chat):

* Requires redeploying the entire backend
* Introduces risk to unrelated components (like streaming)

This reduces:

* Deployment frequency
* System reliability

---

## 5. Problem: Team Scalability

As teams grow:

* Multiple developers work on the same codebase
* Changes overlap

Result:

* Merge conflicts
* Slower development cycles

---

## 6. Problem: Technology Constraints

Different system components have different requirements:

* Streaming → low latency, high performance
* Recommendations → ML models (Python)
* Payments → strict reliability

A monolith forces:

* One technology stack for everything

---

## 7. Architectural Shift: Microservices

Microservices address these problems by:

> Breaking a single backend into multiple independent services

Each service:

* Handles a specific responsibility
* Runs independently
* Can scale independently

---

## 8. System Breakdown (IPL Example)

Instead of one monolith:

```txt 
Auth Service
Streaming Service
Score Service
Recommendation Service
Chat Service
Payment Service
```

Each service is:

* Independently developed
* Independently deployed
* Independently scaled

---

## 9. Independent Scaling (Key Advantage)

Now scaling becomes precise:

```txt 
Streaming → Scale heavily
Auth → Scale moderately
Recommendations → Minimal scaling
```

This results in:

* Efficient resource usage
* Lower cost
* Better performance

---

## 10. Independent Deployment

Each service can be deployed separately:

* Update chat → only deploy chat service
* No impact on streaming or payments

This improves:

* Release speed
* System stability

---

## 11. Fault Isolation

In a monolith:

* One failure can affect the entire system

In microservices:

* Failure is isolated

Example:

* Chat service fails
* Streaming continues working

---

## 12. Technology Flexibility

Each service can use the most suitable technology:

* Streaming → Go / C++
* Recommendations → Python
* APIs → Node.js

This enables:

* Better performance
* Better optimization

---

## 13. New Complexity Introduced

Microservices solve many problems, but introduce new ones:

### 1. Service Communication

Services must communicate over network.

---

### 2. Data Consistency

Each service may have its own database.

---

### 3. Increased Infrastructure Complexity

More services → more management overhead

---

## 14. When Microservices Make Sense

Microservices are suitable when:

* System is large
* Traffic is high
* Teams are growing
* Features are independent

---

## 15. When NOT to Use Microservices

Avoid microservices when:

* Application is small
* Team is small
* System complexity is low

A modular monolith is often better in early stages.

---

## 16. Key Insight

Microservices are not a “better architecture” by default.

They are:

> A solution to specific scaling, deployment, and organizational problems.

---

## 17. Summary

* Monolith limits independent scaling and deployment
* Microservices split system into independent units
* Each service can scale, deploy, and evolve separately
* They introduce flexibility but also complexity
