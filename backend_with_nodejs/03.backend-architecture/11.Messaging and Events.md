# 08. Messaging and Events (Asynchronous Systems)

## Why Systems Move Beyond Synchronous Communication


## 1. Limitation of Synchronous Communication

In synchronous communication:

```txt id="sync-flow"
Service A → Service B → wait → response
```

This creates dependency:

* Service A is blocked until Service B responds
* If Service B is slow or unavailable, Service A is affected

---

## 2. Problem in Real Systems

In high-scale systems, this leads to:

* Increased latency
* Cascading failures
* Reduced system reliability

---

### IPL Scenario

On a platform like JioHotstar:

When a user performs an action:

* Watch match
* Like content
* Send chat message

Multiple services are involved.

If all communication is synchronous:

* One slow service delays the entire flow
* System becomes fragile under load

---

## 3. Need for Asynchronous Communication

To avoid blocking and dependency:

> Systems shift to asynchronous communication

---

## 4. What is Asynchronous Communication

In asynchronous systems:

```txt id="async-flow"
Service A → Message Queue → Service B (processes later)
```

Key difference:

* Service A does not wait
* Service B processes independently

---

## 5. Message Queue Concept

A message queue acts as an intermediary:

* Producer sends message
* Consumer processes it

---

### Flow

```txt id="mq-flow"
Producer → Queue → Consumer
```

---

## 6. Real Example: Order Processing

### Scenario: Food Delivery Platform

On Swiggy:

When a user places an order:

Instead of:

```txt id="sync-order"
Order Service → Payment → Delivery → Notification
```

System uses:

```txt id="async-order"
Order Service → Queue → Multiple services process independently
```

---

### Result

* Faster response to user
* Independent processing
* Better scalability

---

## 7. Pub/Sub Model

Pub/Sub (Publish-Subscribe) is an extension of messaging.

---

### How it works

```txt id="pubsub"
Publisher → Topic → Multiple Subscribers
```

---

### Example

When an order is placed:

* Order event is published
* Multiple services subscribe:

  * Notification service
  * Analytics service
  * Delivery service

---

## 8. Event-Driven Architecture

In event-driven systems:

* Services react to events
* No direct communication required

---

### Example

Event:

```txt id="event"
OrderPlaced
```

Consumers:

* Send notification
* Assign delivery
* Update analytics

---

## 9. Tools Used

Common messaging systems:

* RabbitMQ
* Apache Kafka

---

### Basic distinction

* RabbitMQ → traditional message queues
* Kafka → high-throughput event streaming

---

## 10. Advantages of Asynchronous Systems

### 1. Decoupling

* Services operate independently

---

### 2. Scalability

* Consumers can scale independently

---

### 3. Fault Tolerance

* If one service fails, messages remain in queue

---

### 4. Improved Performance

* No waiting between services

---

## 11. Challenges Introduced

### 1. Eventual Consistency

* Data is not updated instantly

---

### 2. Debugging Complexity

* Harder to trace flow

---

### 3. Message Ordering

* Events may not arrive in sequence

---

## 12. Key Insight

Synchronous systems focus on:

> Immediate response

Asynchronous systems focus on:

> Reliability and scalability

---

## 13. Summary

* Synchronous communication creates dependency between services
* Asynchronous communication removes blocking
* Message queues enable decoupled systems
* Pub/Sub enables multiple consumers
* Event-driven systems improve scalability and flexibility

---
