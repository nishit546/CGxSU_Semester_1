
#  Native Side in React Native

In React Native, **everything you see and interact with** is powered by a combination of JavaScript and native platform code. Understanding the **Native Side** helps you build more powerful apps and know when you need to go beyond pure JavaScript.

---

## What is the Native Side?

React Native apps run in **two worlds** at the same time:

- **JavaScript World** — Where your React components, logic, and state live.
- **Native World** — The actual Android and iOS platforms.

**Real-life Analogy**:  
Think of React Native as a skilled translator (JavaScript) working with two experts (Android & iOS). When JavaScript alone isn’t enough, it asks the native experts for help.

### Native Code Languages
- **Android**: Kotlin (preferred) or Java
- **iOS**: Swift (preferred) or Objective-C

---

### **Simple Example: Using a Native Feature**

```jsx
import { Alert } from 'react-native';

const checkBatteryLevel = async () => {
  // This would use a Native Module under the hood
  // (In real apps, you might use expo-battery or a custom module)
  Alert.alert('Battery', 'This feature needs native code to work properly');
};

const App = () => {
  return (
    <View style={styles.container}>
      <Text>Native Side Example</Text>
      <Button title="Check Battery Level" onPress={checkBatteryLevel} />
    </View>
  );
};
```

---

### Common Reasons to Use the Native Side

- Accessing device hardware (Camera, GPS, Bluetooth, Sensors)
- Using platform-specific UI components
- Integrating third-party native SDKs
- High-performance operations
- Custom animations or rendering

## 1. React Native Architecture

**Description**: React Native uses a hybrid architecture that combines JavaScript with native platform capabilities.

### **What is React Native Architecture?**

At its core, a React Native app consists of three main layers:

1. **JavaScript Layer** — Your React components, business logic, state management, etc.
2. **Bridge / Communication Layer** — How JavaScript talks to the native side.
3. **Native Layer** — Actual Android (Kotlin/Java) and iOS (Swift/Objective-C) code.

### **Visual Architecture**

```text
+---------------------+
|   JavaScript Code   |   ← Your .jsx files
+---------------------+
          │
          ▼
+---------------------+
|  React Native       |
|  Runtime (Hermes)   |
+---------------------+
          │
          ▼
   +-------------+ 
   |   Bridge    |   ← Old Architecture (Async, Serialized)
   |   or JSI    |   ← New Architecture (Direct, Fast)
   +-------------+
          │
          ▼
+---------------------+
|   Native Modules    |
|   & UI Components   |   ← Android / iOS Code
+---------------------+
          │
          ▼
   Device APIs (Camera, GPS, Bluetooth, File System, etc.)
```

---

### **Basic Complete Example – Understanding Layers**

```jsx
import React from 'react';
import { View, Text, StyleSheet, Alert } from 'react-native';

```

### **Key Layers Explained**

| Layer                | Technology                     | Responsibility |
|----------------------|--------------------------------|----------------|
| JavaScript           | Hermes (default)               | UI Logic, State |
| Bridge / JSI         | Communication layer            | JS ↔ Native |
| Native Modules       | Kotlin / Swift                 | Device features |
| Native UI Components | Android Views / UIKit          | Rendering UI |

## 2. Native Modules

**Description**: Native Modules allow JavaScript code to call platform-specific native code (Android & iOS).

### **What are Native Modules?**

Native Modules are like **bridges** that expose native functionality to your JavaScript code. They let you access device hardware, system services, and third-party SDKs that are not available in pure JavaScript.

**Real-life Analogy**:  
Native Modules are like a waiter in a restaurant — you (JavaScript) tell the waiter what you want, and the waiter (Native Module) communicates with the chef (Native Platform) to deliver the result.

---

### **Basic Complete Example (Using Existing Module)**

```jsx
import React, { useState } from 'react';
import { View, Text, Button, StyleSheet, Alert } from 'react-native';
import * as Battery from 'expo-battery';   // Uses Native Module under the hood

const App = () => {
  const [batteryLevel, setBatteryLevel] = useState(null);

  const getBatteryInfo = async () => {
    const level = await Battery.getBatteryLevelAsync();
    const state = await Battery.getBatteryStateAsync();
    
    setBatteryLevel(level);
    
    Alert.alert(
      'Battery Status',
      `Level: ${Math.round(level * 100)}%\nState: ${state}`,
      [{ text: 'OK' }]
    );
  };

  return (
    <View style={styles.container}>
      <Text style={styles.title}>Native Module Example</Text>
      <Text style={styles.info}>
        Battery Level: {batteryLevel ? Math.round(batteryLevel * 100) + '%' : 'Unknown'}
      </Text>
      
      <Button title="Get Battery Info (Native)" onPress={getBatteryInfo} />
    </View>
  );
};

const styles = StyleSheet.create({
  container: { 
    flex: 1, 
    justifyContent: 'center', 
    alignItems: 'center', 
    padding: 20 
  },
  title: { fontSize: 24, fontWeight: 'bold', marginBottom: 20 },
  info: { fontSize: 18, marginVertical: 20 },
});

export default App;
```

---

### **Important Concepts**

| Concept                  | Description                                      | Example Use Cases |
|-------------------------|--------------------------------------------------|-------------------|
| Built-in Native Modules | Provided by React Native or Expo                 | `Alert`, `Share`, `AsyncStorage` |
| Community Modules       | Third-party libraries                            | `expo-camera`, `react-native-ble-plx` |
| Custom Native Modules   | Written by you in Kotlin/Swift                   | Custom SDK integration |
| Turbo Modules           | Modern version (New Architecture)                | Faster & lazy-loaded |

---

### **More Examples**

#### **1. Using Camera (Native Module)**
```jsx
import { Camera } from 'expo-camera';

const takePicture = async () => {
  if (cameraRef.current) {
    const photo = await cameraRef.current.takePictureAsync();
    console.log('Photo taken via Native Camera:', photo.uri);
  }
};
```

#### **2. Bluetooth Example (Third-party)**
```jsx
import { BleManager } from 'react-native-ble-plx';

const manager = new BleManager();

const scanDevices = () => {
  manager.startDeviceScan(null, null, (error, device) => {
    if (device) console.log('Found device:', device.name);
  });

```

#### **3. Custom Native Module (Conceptual)**
```jsx
// In JavaScript
import { NativeModules } from 'react-native';
const { MyCustomModule } = NativeModules;

const result = await MyCustomModule.doHeavyCalculation(1000000);
```

---

### **When to Create Your Own Native Module?**

- You need to use a native SDK not available in JavaScript
- You want maximum performance for heavy computation
- You need low-level access to hardware
- You want to expose custom native UI

**Pro Tip**: Always check if an Expo or community library exists before writing your own native code.

## 4. The Bridge (Old Architecture)

**Description**: The Bridge is the original communication layer between JavaScript and native code in React Native.

### **What is the Bridge?**

The Bridge is a **serialization-based communication system** that allows JavaScript and native code (Android/iOS) to talk to each other.

It was the core of React Native’s architecture for many years. All data passing between JS and Native goes through this bridge in the form of serialized JSON messages.

**Real-life Analogy**:  
Think of the Bridge as a narrow two-way road between two cities (JavaScript and Native). Everything has to be packed into trucks (JSON), driven across the bridge, and unpacked on the other side.

---

### **Basic Visual Example**

```text
JavaScript Code
       │
   Serialized JSON
       │
    ↓ Bridge ↓
       │
Native Code (Kotlin / Swift)
```

---

### **How the Bridge Works (Step by Step)**

1. JavaScript calls a native method
2. The call is serialized into JSON
3. Message is sent across the Bridge (asynchronously)
4. Native side receives and executes the command
5. Result is serialized back to JSON
6. Sent back to JavaScript thread

---

### **Code Example Demonstrating Bridge Usage**

```jsx
import React from 'react';
import { View, Text, Button, StyleSheet, NativeModules, Alert } from 'react-native';

const App = () => {
  const { CalendarModule } = NativeModules; // Example Native Module

  const addEvent = async () => {
    try {
      // This call goes through the Bridge
      await CalendarModule.createCalendarEvent(
        'React Native Meetup', 
        'Tomorrow at 6 PM'
      );
      Alert.alert('Success', 'Event added via Native Bridge!');
    } catch (error) {
      Alert.alert('Error', error.message);
    }
  };

  return (
    <View style={styles.container}>
      <Text style={styles.title}>The Old Bridge Architecture</Text>
      <Text style={styles.subtitle}>
        Every native call goes through the Bridge
      </Text>
      
      <Button title="Add Calendar Event (Via Bridge)" onPress={addEvent} />
    </View>
  );
};

const styles = StyleSheet.create({
  container: { 
    flex: 1, 
    justifyContent: 'center', 
    alignItems: 'center', 
    padding: 20 
  },
  title: { fontSize: 22, fontWeight: 'bold', textAlign: 'center', marginBottom: 12 },
  subtitle: { 
    fontSize: 16, 
    color: '#666', 
    textAlign: 'center', 
    marginBottom: 30,
    lineHeight: 24 
  },
});

export default App;
```

---

### **Limitations of the Bridge**

| Limitation                  | Description                                      | Impact |
|----------------------------|--------------------------------------------------|--------|
| Asynchronous               | All communication is async                       | Delays in UI updates |
| Serialization Overhead     | Data must be converted to JSON                   | Slow for large data |
| Single Threaded            | JavaScript thread can get blocked                | Janky UI |
| Batch Processing           | Messages are processed in batches                | Latency |
| Memory Copying             | Data is copied multiple times                    | High memory usage |

---

### **More Examples of Bridge Communication**

- Calling `AsyncStorage` → Goes through Bridge
- Using `Alert` → Native UI via Bridge
- Accessing `Dimensions` → Native call via Bridge

---

### **Why the Bridge Was a Bottleneck**

- Heavy data transfer (e.g., large lists or images) became slow
- High-frequency updates caused lag
- Startup time was slower due to bridge initialization

This is why the **New Architecture (JSI + Turbo Modules + Fabric)** was introduced.

## 5. JSI (JavaScript Interface)

**Description**: JSI is a modern, high-performance communication layer between JavaScript and native code.

### **What is JSI?**

**JSI (JavaScript Interface)** is a direct interface that allows JavaScript to call native functions **without** going through the traditional Bridge serialization process.

It was introduced as part of React Native’s New Architecture to solve the performance limitations of the old Bridge.

**Real-life Analogy**:  
If the old Bridge was a narrow road with checkpoints (serialization), **JSI** is like building a direct high-speed expressway between JavaScript and Native code.

---

### **How JSI Works (Visual)**

```text
JavaScript (Hermes Engine)
         │
       JSI (Direct Binding)
         │
   Native C++ / Kotlin / Swift
```

No JSON serialization → Much faster execution.

---

### **Basic Example (Conceptual – New Architecture)**

```jsx
// This is how modern Turbo Modules work under the hood with JSI

import React, { useEffect } from 'react';
import { View, Text, StyleSheet } from 'react-native';

const App = () => {
  useEffect(() => {
    // In New Architecture, this call is much faster via JSI
    console.log('JSI-enabled Native Call');
    
    // Example: Calling a Turbo Module directly
    // SomeNativeModule.doFastOperation(data);
  }, []);

  return (
    <View style={styles.container}>
      <Text style={styles.title}>JSI - JavaScript Interface</Text>
      <Text style={styles.description}>
        Direct communication between JavaScript and Native code{'\n'}
        (No more heavy Bridge serialization)
      </Text>
    </View>
  );
};

const styles = StyleSheet.create({
  container: { 
    flex: 1, 
    justifyContent: 'center', 
    alignItems: 'center', 
    padding: 30,
    backgroundColor: '#f8f9fa' 
  },
  title: { 
    fontSize: 26, 
    fontWeight: 'bold', 
    marginBottom: 20,
    textAlign: 'center' 
  },
  description: { 
    fontSize: 17, 
    textAlign: 'center', 
    lineHeight: 26,
    color: '#444' 
  },
});

export default App;
```

---

### **Key Benefits of JSI**

| Benefit                    | Description                                      | Impact |
|---------------------------|--------------------------------------------------|--------|
| Direct Memory Access      | No serialization/deserialization                 | Much faster |
| Synchronous Calls         | Possible where needed                            | Better for performance-critical code |
| Lower Overhead            | Less copying of data                             | Reduced memory usage |
| Foundation for New Arch   | Powers Turbo Modules & Fabric                    | Future-proof |

---

### **More Examples & Context**

#### **1. JSI in Action (Behind the Scenes)**
- Calling a Turbo Module method
- Fabric Renderer updating UI
- High-frequency sensor data
- Complex animations with Reanimated 3

#### **2. Performance Comparison**
```js

// New JSI + TurboModule (Faster)

```

### **Current Status (2026)**

- JSI is the foundation of the **New Architecture**
- Many core modules already use JSI
- Full adoption is increasing rapidly
- You get better performance even if you don’t write native code

**Pro Tip**: You don’t need to understand JSI deeply to benefit from it — enabling the New Architecture automatically gives you these improvements.

## 6. Turbo Modules

**Description**: Turbo Modules are the modern replacement for the old Native Modules system in React Native’s New Architecture.

### **What are Turbo Modules?**

**Turbo Modules** are a new, high-performance way to expose native functionality to JavaScript. They are built on top of **JSI** and solve many limitations of the old Bridge-based Native Modules.

They are **lazy-loaded**, meaning they are only initialized when actually needed, leading to faster app startup times.

---

### **Basic Complete Example (Conceptual – New Architecture)**

```jsx
import React, { useEffect, useState } from 'react';
import { View, Text, StyleSheet, Button } from 'react-native';
// TurboModuleRegistry is part of the New Architecture
import { TurboModuleRegistry } from 'react-native';

const App = () => {
  const [result, setResult] = useState(null);

  const callTurboModule = async () => {
    try {
      // In New Architecture, this is a Turbo Module
      const CustomModule = TurboModuleRegistry.get('CustomDeviceModule');
      
      if (CustomModule) {
        const data = await CustomModule.getDeviceInfo();
        setResult(data);
        console.log('Turbo Module Result:', data);
      }
    } catch (e) {
      console.error('Turbo Module Error:', e);
    }
  };

  return (
    <View style={styles.container}>
      <Text style={styles.title}>Turbo Modules</Text>
      <Text style={styles.subtitle}>
        Modern, lazy-loaded, JSI-powered native modules
      </Text>

      <Button title="Call Turbo Module" onPress={callTurboModule} />

      {result && (
        <Text style={styles.result}>
          Device Info: {JSON.stringify(result, null, 2)}
        </Text>
      )}
    </View>
  );
};

const styles = StyleSheet.create({
  container: { 
    flex: 1, 
    justifyContent: 'center', 
    alignItems: 'center', 
    padding: 20 
  },
  title: { fontSize: 24, fontWeight: 'bold', marginBottom: 8 },
  subtitle: { 
    fontSize: 16, 
    color: '#555', 
    textAlign: 'center', 
    marginBottom: 30 
  },
  result: { 
    marginTop: 20, 
    fontSize: 14, 
    backgroundColor: '#f0f0f0', 
    padding: 15, 
    borderRadius: 10 
  },
});

export default App;
```

---

### **Key Advantages of Turbo Modules**

| Advantage                  | Description                                      | Benefit |
|---------------------------|--------------------------------------------------|--------|
| Lazy Loading              | Modules load only when used                      | Faster app startup |
| JSI-Based                 | Direct JavaScript ↔ Native communication        | High performance |
| Type Safety               | Better TypeScript support                        | Fewer runtime errors |
| Smaller Memory Footprint  | Less overhead than old modules                   | Better efficiency |
| Asynchronous by Default   | But supports sync calls when needed              | Flexible |

---

### **More Examples**

#### **1. Using Built-in Turbo Modules**
```jsx
// Example: New Architecture version of common modules
import * as Clipboard from 'expo-clipboard'; // Often uses TurboModule under the hood
```

#### **2. Creating a Turbo Module (Conceptual)**
```js
// JavaScript side
const MyTurboModule = TurboModuleRegistry.getEnforcing('MyTurboModule');

const result = await MyTurboModule.processLargeData(bigArray);
```

---

### **Old Native Modules vs Turbo Modules**

| Feature                    | Old Native Modules          | Turbo Modules               |
|---------------------------|-----------------------------|-----------------------------|
| Loading                   | Eager (at startup)          | Lazy (on demand)            |
| Communication             | Bridge (JSON)               | JSI (Direct)                |
| Performance               | Good                        | Excellent                   |
| Startup Time              | Slower                      | Faster                      |
| Code Complexity           | Higher boilerplate          | Cleaner                     |

---

### **Current Status**

- Turbo Modules are a core part of the **New Architecture**
- Many official and community libraries have migrated
- You benefit automatically when you enable the New Architecture

**Pro Tip**: Even if you're not writing custom native code, enabling Turbo Modules improves your app’s startup time and overall performance.

## 7. Fabric

**Description**: Fabric is React Native’s new rendering system (part of the New Architecture).

### **What is Fabric?**

**Fabric** is the new UI rendering engine in React Native. It replaces the old rendering system and works together with **JSI** and **Turbo Modules** to deliver significantly better performance and new capabilities.

It is also known as the **New Renderer**.

---

### **Basic Visual Explanation**

```text
React Components (JS)
         │
   React Reconciler
         │
      Fabric Renderer   ← New, High-Performance
         │
   Native UI Components (Android Views / iOS UIView)
```

---

### **Complete Example – Understanding the Difference**

```jsx
import React, { useState } from 'react';
import { View, Text, StyleSheet, Button, ScrollView } from 'react-native';

        title="Trigger Heavy Update"

  );
```

### **Key Benefits of Fabric**

| Benefit                        | Description                                      | Real Impact |

## 9. When Do You Need Native Code?

**Description**: Not every feature requires writing native code. Knowing when to go native is an important skill.

### **What Does “Native Code” Mean?**

It means writing Kotlin/Java (Android) or Swift/Objective-C (iOS) code to extend React Native’s capabilities.

---

### **When You Should Consider Native Code**

Here’s a clear decision guide:

| Situation                              | Recommendation                  | Alternative |
|---------------------------------------|----------------------------------|-----------|
| Basic device features (Camera, GPS)   | Use Expo / Community libraries   | No native needed |
| Advanced Camera / AR                  | Custom Native Module or UI       | Sometimes required |
| Bluetooth / NFC                       | Usually needs native             | Libraries available |
| Custom Hardware Integration           | Native Module                    | Required |
| Payment Terminals / Enterprise SDKs   | Native Module                    | Required |
| High-performance graphics             | Native UI Component              | Often needed |
| Complex Custom UI                     | Native UI Component              | Consider Fabric |
| Simple UI / Logic                     | Pure JavaScript                  | Preferred |

---

### **Basic Decision Example**

```jsx
import React from 'react';
import { View, Text, StyleSheet, Button, Alert } from 'react-native';

const App = () => {
  const checkIfNativeNeeded = (feature) => {
    const nativeFeatures = [
      'advanced-bluetooth',
      'nfc-payment',
      'custom-thermal-printer',
      'ar-experience'
    ];

    if (nativeFeatures.includes(feature)) {
      Alert.alert(
        'Native Code Required', 
        `Feature "${feature}" needs custom native implementation.`
      );
    } else {
      Alert.alert('JavaScript Only', `Feature "${feature}" can be done with JS + libraries.`);
    }
  };

  return (
    <View style={styles.container}>
      <Text style={styles.title}>Do I Need Native Code?</Text>
      
      <Button 
        title="Check Camera Access" 
        onPress={() => checkIfNativeNeeded('camera')} 
      />
      <Button 
        title="Check NFC Payment" 
        onPress={() => checkIfNativeNeeded('nfc-payment')} 
      />
      <Button 
        title="Check AR Experience" 
        onPress={() => checkIfNativeNeeded('ar-experience')} 
      />
    </View>
  );
};

const styles = StyleSheet.create({
  container: { 
    flex: 1, 
    justifyContent: 'center', 
    alignItems: 'center', 
    padding: 20,
    gap: 12 
  },
  title: { fontSize: 24, fontWeight: 'bold', marginBottom: 30, textAlign: 'center' },
});

export default App;
```

### **Detailed Scenarios**

**You Probably DON’T Need Native Code If:**
- You’re using Expo (managed workflow)
- The feature exists in `expo-*` packages
- Community libraries like `react-native-maps`, `expo-camera` are sufficient
- You only need basic device info

**You Likely NEED Native Code When:**
- Integrating proprietary enterprise SDKs
- Working with specialized hardware (printers, scanners, medical devices)
- Needing very high-performance custom rendering
- Implementing unique platform-specific gestures or UI
- Working with low-level system APIs

---

### **Best Practices**

- **Always check first**: Search for existing Expo or community libraries
- Prefer **Expo Modules** when possible (they simplify native development)
- Keep native code minimal and well-separated
- Document your native modules clearly
- Test thoroughly on both Android and iOS

---

**Pro Tip**: In 2026, with the New Architecture and powerful libraries, **most apps can be built with little to no custom native code**.
